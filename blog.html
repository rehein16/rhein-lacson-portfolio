What I Learned Building My First Real Project

Feb 2026 · 5 min read

Building Aid4Odette wasn't just an academic exercise — it was my first real encounter with the gap between writing code and designing an experience. Here's what the process taught me about purpose-driven development.

The Brief That Changed Everything

When I started working on Aid4Odette, I thought it would be straightforward: build a donation portal, add a form, done. What I didn't anticipate was how much the emotional weight of the subject matter would influence every design decision. This wasn't just a school project — real people would use this to help real families affected by the typhoon. That pressure made me think harder than any grade ever did.

"Design isn't just what it looks like. Design is how it works." — Steve Jobs

Learning to Design for Trust

Donation platforms live or die by trust. Users need to feel confident that their money is going somewhere real. That meant clean typography, transparent fund allocation breakdowns, and a checkout flow that felt secure — not like a sketchy phishing page. I redesigned the donation form four times before it felt right. Each iteration taught me something new about clarity, hierarchy, and friction reduction.

Technical Wins and Losses

I pushed my CSS layout skills further than I ever had — the hero section uses a combination of CSS Grid and absolute positioning that took me two full days to nail. I also learned the hard way that audio autoplay is blocked by default in modern browsers and requires explicit user interaction to trigger. That wasted almost half a day before I found the fix. Debugging real-world browser behavior is a completely different skill from writing clean tutorial code.

What I'd Do Differently

I'd plan the component structure before writing a single line of CSS. I jumped straight into styling and ended up with a tangled mess of overrides halfway through. Starting with a rough wireframe in Figma first would have saved me hours. That's now a non-negotiable step in my process.

Key Takeaways

Every project teaches you something you didn't know you needed to know. Aid4Odette taught me that good UI is inherently empathetic — you're always designing for a human being on the other side of the screen, and that responsibility should shape every pixel you place.

CSS Grid vs Flexbox: When to Use Which

Jan 2026 · 4 min read

After building three full projects, I finally have a concrete mental model for when to reach for Grid and when Flexbox is the right tool. Spoiler: it's about one dimension versus two.

The Core Distinction

Flexbox is one-dimensional: it controls layout along a single axis — either a row or a column, never both at the same time. Grid is two-dimensional: it manages rows and columns simultaneously. That single sentence answers about 80% of "which one do I use?" questions. Once it clicked for me, the confusion mostly went away.

Use Flexbox When...

You're aligning items in a navbar, distributing buttons in a row, centering content inside a card, or handling a tag list that wraps naturally. Anything that's fundamentally "items in a line" — reach for flex. It's also ideal when you need to push one item to the far end of a container, like a logo on the left and a CTA button on the right. That's a flex job, not a grid job.

Use Grid When...

You're building a full page layout, a card grid, a magazine-style section, or anything where you need explicit control over both rows and columns at the same time. Grid really shines when items need to align across two axes — like a dashboard where cards in different rows still need to line up vertically. Grid also makes complex responsive layouts dramatically simpler with just grid-template-columns: repeat(auto-fit, minmax(240px, 1fr)).

A Gotcha I Hit

Early on I kept trying to use Flexbox for two-dimensional card grids and wondering why everything looked off when the screen resized. The cards would grow unevenly, or the last row would be misaligned. The moment I switched to Grid with auto-fit, everything snapped into place. That's a mistake I've never repeated since.

The Real Answer

Use both — together. They're not competitors, they're complements. A Grid layout with Flexbox inside each grid cell is the standard pattern in almost every modern interface, including this portfolio. Grid owns the macro structure; Flexbox handles the micro alignment inside each piece.

Preparing for OJT: What I'm Doing Right Now

Dec 2025 · 3 min read

OJT season is approaching and I've been intentional about how I'm preparing. Here's a breakdown of what I'm building, learning, and improving before I step into a professional environment.

Building Real Things, Not Just Tutorials

The best preparation for a professional environment is doing professional-quality work. That's why I deliberately stopped following tutorials and started building from scratch — Aid4Odette, Wings 2 Go, and Velour all had real constraints, real stakeholders in mind, and real deployment challenges. There's a gap between finishing a tutorial and shipping something that actually works in a browser on someone else's device. I've been living in that gap, and it's the most valuable place I've ever learned.

Being Selective About Certifications

I'm not collecting certifications for the sake of a longer resume. I'm targeting ones that fill genuine gaps in my knowledge. The freeCodeCamp Back End Development certification pushed me to understand server-side architecture, databases, and APIs in ways I'd been avoiding as a front-end focused developer. Each cert I pursue has a reason behind it — a weakness I've identified, a skill that kept coming up in job postings, or a technology I struggled to use confidently.

This Portfolio as a Project

Building this portfolio has been one of the most valuable projects I've completed — not because it looks good (though I hope it does), but because it forced me to think critically about how I present myself, what I've actually accomplished, and where I genuinely want to go. Writing the "About" section alone took longer than building most of the layout. Articulating your own value is a skill, and it's one I hadn't practiced enough.

What's Next

React is my current focus. I want to be genuinely comfortable building components, managing state, and thinking in a component-based architecture before I enter OJT. After that: TypeScript for type safety, then a full backend project with authentication and a real database. The goal isn't to know everything — it's to be confident enough to learn fast once I'm inside a real team.
